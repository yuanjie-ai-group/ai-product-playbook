# Stage 1：看懂用户，做对产品（1~3 个月）

> **核心转变**：从"怎么实现"到"该不该做"
>
> 📁 [`01-产品思维/`](./01-产品思维/) · [`02-AI产品专项/`](./02-AI产品专项/)

## 1.1 用户研究

**核心方法**：

- **用户访谈** — 不是问"你想要什么"，而是问"你上次遇到这个问题时怎么做的"
- **用户画像** — 角色 + 场景 + 痛点 + 目标，不是拍脑袋编故事
- **用户旅程地图** — 从第一次听说产品到深度使用的全过程，标出每个阶段的情绪和断点
- **反馈收集** — 用户嘴上说的 ≠ 实际需要的，观察行为比听意见更重要

## 1.2 需求分析

**需求三问**（每个需求必问）：

> 给谁用？解决什么问题？怎么衡量成功？

**表面需求 vs 深层需求**：用户说"我想要更快的马" → 深层需求是"更快到达目的地"

**需求优先级框架**：

| 框架 | 公式 | 适用场景 |
|------|------|---------|
| **RICE** | Reach × Impact × Confidence / Effort | 数据充分时的量化排序 |
| **ICE** | Impact × Confidence × Ease | 快速粗排 |
| **MoSCoW** | Must / Should / Could / Won't | 版本规划时划分边界 |

**PRD 核心结构**：背景与目标 → 用户故事 → 功能需求 → 非功能需求 → 验收标准 → 里程碑

## 1.3 产品设计

- **信息架构** — 用户能用 3 步以内找到想要的功能
- **MVP** — 用最小代价验证核心假设，不是做一个"简陋版本"，而是做"刚好够验证"的版本
- **交互设计** — 工程师容易忽略的点：空状态、加载态、错误态、边界态

## 1.4 竞品分析

**四维拆解框架**：

| 维度 | 核心问题 |
|------|---------|
| **功能** | 做了什么功能？哪些是核心差异化功能？ |
| **体验** | 上手体验如何？Aha Moment 在哪？ |
| **商业** | 怎么收费？目标客户是谁？ |
| **技术** | 用了什么模型/架构？技术壁垒在哪？ |

## 1.5 AI 产品设计模式

| 模式 | 典型产品 | 核心交互 | 设计要点 |
|------|---------|---------|---------|
| **对话式** | ChatGPT、Claude | 自然语言 → 文本 | 上下文管理、对话引导、输出格式控制 |
| **生成式** | Midjourney、Suno | 描述 → 创意内容 | 参数暴露程度、迭代编辑、质量一致性 |
| **辅助式** | Copilot、Grammarly | 用户操作 → 实时建议 | 触发时机、打断容忍度、采纳率 |
| **Agent式** | Devin、Manus | 目标 → AI 自主完成 | 透明度、中间状态可控、容错回退 |
| **搜索增强式** | Perplexity | 问题 → 带引用的回答 | 来源可信度、答案结构化、追问引导 |

## 1.6 AI 产品独特挑战

| 挑战 | 本质 | 应对策略 |
|------|------|---------|
| **非确定性输出** | 同样输入 ≠ 同样结果 | 设计"重新生成"机制，用 temperature 控制 |
| **质量难以保证** | 无法 100% 测试覆盖 | 建立评估基准 + 人工抽检 + 用户反馈闭环 |
| **成本与质量矛盾** | 好模型 = 高成本 | 分层调用：简单任务小模型，复杂任务大模型 |
| **用户预期管理** | 高估或低估 AI 能力 | 引导式上手 + 能力边界提示 + 示例展示 |
| **技术底座快速变化** | 模型几个月一迭代 | 抽象模型层，可快速切换底层模型 |

**Prompt 产品化**：系统 Prompt 不是一次写完，而是产品化管理 — 版本控制 → 模板库 → A/B 测试 → 效果评估

# Stage 2：驱动增长，跑通数据（2~4 个月）

> **核心转变**：从"做了功能"到"看到数据涨了"
>
> 📁 [`03-增长与运营/`](./03-增长与运营/) · [`04-数据驱动/`](./04-数据驱动/)

## 2.1 增长框架

**AARRR 漏斗** — 每个环节都有对应指标和打法：

```
Acquisition  →  Activation  →  Retention  →  Revenue  →  Referral
用户从哪来       首次体验价值     持续回来用      愿意付钱       愿意推荐
(渠道、CAC)    (Aha Moment)    (留存曲线)     (转化率)      (NPS、K因子)
```

**北极星指标** — 找到最能代表产品价值的那个数字。好的北极星指标 = 反映用户获得的核心价值 + 可被团队行动影响 + 与收入正相关

**增长飞轮**：内容 → 用户 → 数据 → 产品优化 → 更多用户 → 更多内容（正循环）

**PMF 验证标准**：如果明天关掉产品，超过 40% 的用户会"非常失望"

## 2.2 获客渠道

| 渠道 | 优势 | 关键指标 |
|------|------|---------|
| **内容营销** | 长尾流量、建立专业信任 | 文章引流量、SEO 排名 |
| **社区运营** | 种子用户、口碑传播 | 社区活跃度、UGC 量 |
| **开发者生态** | API 带来长尾增长 | 开发者注册数、API 调用量 |
| **Product-Led Growth** | 产品自带传播属性 | 自然注册率、邀请转化率 |
| **付费投放** | 快速验证、可量化 | CAC、ROI |

### AI 产品增长的特殊性

- **体验即增长** — Aha Moment 通常来自第一次使用
- **内容是武器** — AI 生成的内容本身可以成为传播素材
- **API 是渠道** — 开发者生态可以带来长尾增长
- **口碑是关键** — AI 产品质量波动大，口碑管理尤为重要

## 2.3 SEO 与 GEO

> 详见 [`03-增长与运营/内容运营/SEO与GEO优化策略.md`](./03-增长与运营/内容运营/SEO与GEO优化策略.md)

| | SEO（传统搜索优化） | GEO（AI 搜索优化） |
|---|---|---|
| **优化对象** | Google/Baidu | ChatGPT、Perplexity、AI Overview |
| **核心目标** | 排名靠前 | 被 AI 引用/推荐 |
| **关键因素** | 关键词、外链、性能 | 内容权威性、结构化数据、可引用性 |
| **策略重点** | 关键词密度、Topic Cluster | 开头直给结论、FAQ Schema、实体权威性 |

**GEO 核心洞察**：AI 搜索不按排名展示，而是生成答案并引用来源。目标是成为被引用的信源——"可引用性"比"可点击性"更重要。

**必建页面**：`/about`（实体信息）、`/glossary`（术语表）、`/comparison`（方案对比）、`/pricing`（清晰定价）、`/cases`（成功案例）

## 2.4 指标体系

**核心指标 vs 虚荣指标** — 注册数是虚荣指标，活跃用户数是核心指标

**指标层级**：

```
北极星指标（1 个）
  └── 一级指标（3~5 个，对应 AARRR 各环节）
        └── 二级指标（每个一级拆解出 2~3 个可执行的子指标）
```

## 2.5 A/B 测试

**完整闭环**：假设 → 方案 → 确定样本量 → 执行 → 等待统计显著 → 分析 → 决策

**常见陷阱**：
- **辛普森悖论** — 分组看都好，合并看反而差
- **偷看数据** — 还没达到样本量就下结论
- **多重比较** — 同时测太多变量，假阳性率飙升

**AI 产品 A/B 测试特殊性**：输出非确定性 → 需要更大样本量 + 更关注用户满意度而非点击率

## 2.6 用户行为分析

| 分析方法 | 回答什么问题 | 工具 |
|---------|------------|------|
| **漏斗分析** | 转化率在哪里断裂？ | PostHog / Mixpanel |
| **路径分析** | 用户实际怎么使用产品？ | PostHog |
| **留存分析** | 用户多久后不再回来？ | Cohort 表 |
| **流失分析** | 用户为什么走了？ | 流失前行为回溯 |

**埋点设计原则**：事件（做了什么）+ 属性（在什么条件下做的） → 提前规划，否则后补成本极高

# Stage 3：商业判断，赚到钱（4~6 个月）

> **核心转变**：从"做一个好产品"到"做一个好生意"
>
> 📁 [`05-商业与战略/`](./05-商业与战略/)

## 3.1 商业模式

**AI 产品常见模式**：

| 模式 | 适用场景 | 优劣 |
|------|---------|------|
| **SaaS 订阅** | 持续使用的工具类产品 | 收入可预测，但需要持续留存 |
| **API 按量计费** | 开发者产品、B2B | 与使用量正相关，但收入波动 |
| **免费增值** | 需要快速获客的产品 | 用户基数大，但转化率通常 < 5% |
| **项目制交付** | AI 技术服务 | 单笔收入高，但不可规模化 |

**单位经济模型** — 三个数字决定生死：

| 指标 | 公式 | 健康标准 |
|------|------|---------|
| **LTV**（用户生命周期价值） | ARPU × 平均使用月数 | — |
| **CAC**（用户获取成本） | 营销总花费 / 新增用户数 | — |
| **LTV / CAC** | — | > 3 才健康 |

**AI 产品成本结构**：推理成本（Token）+ 带宽 + 存储，模型选择直接决定毛利率

## 3.2 定价策略

| 策略 | 思路 | AI 产品适用性 |
|------|------|-------------|
| **价值定价** | 按客户获得的价值定价 | 最推荐，但需量化价值 |
| **成本加成** | 成本 × 倍数 | 适合项目制，对 SaaS 不适用 |
| **竞争定价** | 参考竞品价格 | 适合红海市场 |

**免费版 vs 付费版划分原则**：免费版让用户感受到核心价值 → 付费版解锁效率/额度/高级功能

**AI 定价特殊性**：Token 成本随用量波动，重度用户可能亏钱 → 需要设置用量上限或阶梯定价

## 3.3 市场定位

- **定位理论核心**：占据用户心智中的一个位置（"xxx 领域最好的 AI 工具"）
- **目标市场选择**：TAM（总市场）→ SAM（可服务市场）→ SOM（可获得市场）
- **差异化竞争**：不要在巨头擅长的维度竞争，找到自己的差异化切入点

## 3.4 财务管理

> 详见 [`05-商业与战略/财务管理/初创公司财务管理框架.md`](./05-商业与战略/财务管理/初创公司财务管理框架.md)

**现金跑道** = 银行余额 / 月均净支出（> 12 月安全，6~12 月警戒，< 6 月危险）

**项目报价公式**：`报价 = 预估人天 × 人天单价 × 风险系数`（简单项目 1.2x，复杂 1.5x，高不确定性 2.0x）

**利润率基准**：标准化交付 50~60%，定制开发 30~40%，纯外包转包 10~20%

**关键财务指标**：

| 指标 | 健康值 |
|------|--------|
| 现金跑道 | > 6 个月 |
| 项目毛利率 | > 30% |
| 回款周期 | < 45 天 |
| 人效比（月收入/人数） | > 3 万/人 |
| 客户集中度 | 最大客户 < 40% |

**AI 工具成本优化**：简单任务用小模型、批量任务用 Batch API（5折）、重复调用用 Prompt 缓存（降 90%）、非敏感任务考虑本地部署

## 3.5 法务合规

> 详见 [`05-商业与战略/法务合规/初创公司法务合规要点.md`](./05-商业与战略/法务合规/初创公司法务合规要点.md)

**必备合同**：

| 合同 | 关键条款 |
|------|---------|
| **项目开发合同** | 范围（附需求文档）、验收标准、付款节点、变更流程、违约责任 |
| **劳动合同** | 知识产权归属、竞业限制（必须付补偿金否则无效） |
| **NDA** | 保密范围、期限、违约金 |
| **合伙人协议** | 出资比例、分红、退出机制、决策权 |

**知识产权**：
- 商标尽早注册（第 9 类软件 + 第 42 类技术服务），官费 270 元/类
- 每个独立产品申请软著（免费，1~2 个月，可享税收优惠 + 招投标加分）
- 外包代码**默认归开发方**，合同必须约定归属
- 注意开源 License 兼容性（GPL 有传染性）

**数据合规**（AI 产品）：个人信息保护法（告知同意、最小必要）+ 生成式 AI 管理办法（内容审核、标注 AI 生成）

**劳动合规要点**：入职 30 天内签合同（否则双倍工资）、社保入职即缴、竞业限制必须按月付补偿金

## 3.6 客户关系管理

> 详见 [`05-商业与战略/客户关系管理/客户关系管理CRM.md`](./05-商业与战略/客户关系管理/客户关系管理CRM.md)

**客户生命周期**：

```
线索 → 商机 → 签约 → 交付 → 验收 → 维护 → 复购/转介绍
```

**商机漏斗转化率基准**：线索→初步沟通 50% → 方案报价 60% → 商务谈判 70% → 签约 80%

**项目健康度信号**：

| 危险信号 | 含义 | 应对 |
|---------|------|------|
| 客户不回消息 | 优先级下降或有顾虑 | 主动上门 |
| 频繁变更需求 | 需求不明确 | 冻结需求，签变更单 |
| 验收拖延 | 可能想拖延付款 | 合同约定"默认验收"条款 |

**客户分级**：A 级（年 >50 万，专人对接）→ B 级（10~50 万，定期回访）→ C 级（<10 万，被动响应）

**复购关键**：交付超预期 + 保持联系 + 主动提供新价值 + 降低决策成本

**初创阶段工具建议**：飞书多维表格管商机 + 微信维护关系，团队超 10 人再上专业 CRM

### 合伙人视角的商业原则

- **现金流 > 利润 > 收入** — 初创公司死于现金流断裂
- **聚焦 > 全面** — 找到一个切入点打穿，而不是铺开做
- **壁垒思维** — 持续思考"别人抄我们怎么办"

# Stage 4：带队决策，持续修炼

> **核心转变**：从个人贡献者（IC）到为整条业务线负责
>
> 📁 [`06-团队与领导力/`](./06-团队与领导力/) · [`08-认知升级/`](./08-认知升级/)

## 4.1 团队搭建

- **招聘策略** — 初创团队找"对的人"而不是"最强的人"：价值观匹配 > 技能匹配
- **激励机制** — 股权（长期绑定）+ 期权（未来激励）+ 成长空间（最容易被忽略）
- **团队文化** — 文化不是写在墙上的标语，是创始人日常行为的映射

## 4.2 跨部门协作

| 场景 | 工程师出身的优势 | 关键方法 |
|------|----------------|---------|
| **产研协同** | 能判断方案可行性和成本 | 需求评审时直接给出技术约束 |
| **向上管理** | 能用数据说服 CEO/投资人 | 结论先行 + 数据支撑 + 风险提示 |
| **向下赋能** | 能直接 review 代码和架构 | 授权 + 辅导，不是替他做 |
| **冲突管理** | 理解双方的技术和产品诉求 | 回到用户价值判断，不是"谁赢" |

## 4.3 战略执行

- **OKR** — O（目标）定方向，KR（关键结果）定量化标准，每季度复盘
- **产品路线图** — 按季度规划，月度迭代，不要做超过 6 个月的详细计划
- **资源分配** — 80% 做确定性高的事，20% 做探索性尝试
- **复盘** — 事实（发生了什么）→ 归因（为什么）→ 行动（下次怎么改）

## 4.4 合伙人修炼

**股权架构**：创始人 > 51%（最好 > 67%）、合伙人股权 4 年分期兑现（Vesting）、预留 10~20% 期权池

**决策机制**：

| 决策类型 | 谁决定 |
|---------|-------|
| 日常经营 | CEO/总经理 |
| 重大事项（融资、大合同、关键人事） | 合伙人一致同意 |
| 根本性变更（增资、转让股权、解散） | 股东会 2/3 以上 |

**心态修炼**：承受不确定性和压力、全局视角（不只看自己负责的部分）、信息不完整时也要做决策

## 4.5 思维模型

| 模型 | 核心思想 | 什么时候用 |
|------|---------|-----------|
| **第一性原理** | 回到最基本的事实，而不是类比 | 做出违反直觉但正确的决策 |
| **系统思维** | 看到反馈回路和非线性效应 | 分析业务飞轮和瓶颈 |
| **逆向思维** | 从"怎么会失败"反推"该怎么做" | 风险评估、产品防御 |
| **概率思维** | 在不确定性中做决策 | 投资判断、优先级排序 |
| **杠杆思维** | 找到投入产出比最高的事 | 时间分配、资源分配 |

## 4.6 沟通表达

- **金字塔原理** — 结论先行 → 分论点 → 事实支撑（工程师最常犯错：从过程讲起）
- **讲故事** — 把复杂的事说简单，用户故事 > 技术方案
- **说服力** — 说服不是"你应该"，而是"对你有什么好处"

### 从工程师到合伙人的关键转变

| 维度 | 工程师 | 合伙人 |
|------|--------|--------|
| 关注点 | 代码质量、系统性能 | 公司发展、团队成长 |
| 成就感 | 解决技术难题 | 团队和产品的成功 |
| 时间分配 | 80% 写代码 | 30% 技术 + 30% 管理 + 40% 战略 |
| 决策方式 | 数据和逻辑 | 数据 + 直觉 + 判断力 |
| 责任范围 | 自己的模块 | 整条业务线甚至整个公司 |

## 核心原则

1. **实战优先** — 所有内容必须结合元境实际产品，不做纯理论
